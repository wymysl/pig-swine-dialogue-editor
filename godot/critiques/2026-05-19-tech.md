# Code & Technical Health — Hostile Critique, 2026-05-19

## Verdict
Save-shape policy is enforced for additive growth but nothing tests the v15→v16 step end-to-end, and no test gates dialogue-catalogue validation failures, so a typo in JSON pushes through smoke green. The signal bus is grown for ceremony — at least four declared signals (`case_folder_acquired`, `case_folder_toggled`, `case_folder_fragment_added`, `judge_skepticism_raised`) emit into the void in production. BlueBinder and BattleController each carry their own packet-scoring engine; the two will drift apart the moment one is patched. Tooling (`voice_audit.py`, `verify_dialogue_roundtrip.js`) is hopeful: there is no CI, no pre-commit, no Makefile, and the smoke aggregator that "passed" for 45 sessions was a no-op skeleton (BUILD_NOTES 2026-05-16). The autoload list disagrees with documentation — `BinderUI` is a fifth autoload that AGENTS.md does not list.

## Findings

### F1 — Smoke test cannot fail on dialogue-catalogue errors
**Flaw:** `tests/test_smoke.gd:23-25` awaits one `process_frame`, then `quit(0)` unconditionally. `DialogueRunner._validate_catalogue` (`scripts/autoload/dialogue_runner.gd:447-533`) is built to push_error on duplicate state ids, unknown speakers, unresolved flag paths, missing content, and legacy `line:` keys — but `push_error` does not change exit status. A renamed `chapter1` flag, a typo in `trigger`, or a `once:true` id collision ships green through smoke + test_runner.
**Why this fails:** "Fail loud" (AGENTS.md Rule 12) is undermined when the loudest assertion in the codebase cannot fail a build. Validation that does not gate a check is documentation.
**Remediation:** Add an error-capture pass to `test_smoke.gd` — instantiate `Main.tscn`, wait one frame, then exit non-zero if any `push_error` was emitted during boot. The simplest implementation is a `Logger` autoload that hooks `Engine.print_error_messages` paths and counts; a less invasive one is parsing the `--log-file` output that test_runner already produces and failing when boot lines contain `"DialogueRunner: "` errors.
**Severity:** S1

### F2 — v15→v16 save migration has no dedicated test
**Flaw:** `tests/` ships `test_save_migration_v7_v8.gd` through `v14_v15`, then jumps to `v16_v17` — `v15_v16` is absent. `save.gd:342-345` is the only place that adds `chapter1.murrow_choice`. Its behaviour is only exercised indirectly when v14 fixtures pass through `migrate_save(...)` and run to current. A regression that drops the key from the v15→v16 branch would be caught only by `test_chapter1_v17_flag_coverage.gd::_test_v15_v16_regression`, which checks the final shape — not the per-step migration boundary.
**Why this fails:** Save policy in `AGENTS.md` §Save And State Policy requires a migration test per version step. v15→v16 is the one gap in an otherwise-complete sequence.
**Remediation:** Add `tests/test_save_migration_v15_v16.gd`, modelled after `test_save_migration_v14_v15.gd`: T1 asserts `SAVE_VERSION >= 16`; T2 feeds a v15 fixture (no `murrow_choice`) through `migrate_save(..., 15)` and asserts the key is added as `""`; T3 asserts pre-existing v15 keys survive; T4 asserts idempotency.
**Severity:** S2

### F3 — Tooling invariants are not wired to any hook
**Flaw:** Repo has no `.github/`, no `.pre-commit-config.yaml`, no `Makefile`, no `pre-commit` git hook (`.git/hooks/` contains only `.sample` files). `tools/voice_audit.py` and `tools/verify_dialogue_roundtrip.js` run only when someone remembers. AGENTS.md command list mentions `voice_audit.py` but treats it as a manual artifact.
**Why this fails:** Voice-reference Rule A/B compliance and dialogue JSON pretty-print contract are both load-bearing for "content is data" architecture. A drift-detection script that drifts in invocation is not a contract — it is a hope.
**Remediation:** Add a Makefile target `make verify` that runs `python tools/voice_audit.py godot/data/voice_references/`, `node tools/verify_dialogue_roundtrip.js`, `godot --headless --path godot --script tests/test_smoke.gd`, and `godot --headless --path godot --script tests/test_runner.gd`. Add a `.git/hooks/pre-commit` (or commit `tools/hooks/pre-commit` and a `make install-hooks` step) that runs `make verify` on commits touching `godot/data/dialogues/` or `godot/data/voice_references/`. Wire the same target into a single-job `.github/workflows/verify.yml`.
**Severity:** S2

### F4 — Two packet-scoring engines, one game
**Flaw:** `scripts/systems/battle/battle_controller.gd::evaluate_packet_submission` (lines 448-523) and `scripts/ui/blue_binder.gd::evaluate_packet` (lines 203-255) each compute a packet's `proposed_frame`, `judicial_patience` start, decoy frame list, remedy-driven `overbroad_remedy`, and trust delta. The two implementations use different priority orders (`DECOY_FRAME_PRIORITY` in BattleController is `incapacity, standing, notice, substantive, overbroad`; `FRAME_PRIORITY` in BlueBinder is `incapacity, overbroad, standing, notice, substantive`), different patience deltas (BattleController subtracts a `support_patience_delta` for under-supported packets; BlueBinder does not), and different gates (incapacity is gated behind `halina_met` in BlueBinder but unconditional in BattleController's selection).
**Why this fails:** Single-writer discipline (AGENTS.md §Save And State Policy "One system writes; others read") applies to packet evaluation too — these two functions read the same state and produce divergent answers. The UI score the player sees on apply_packet_assessment will not match the score the court round consumes via consume_assembled_packet.
**Remediation:** Extract packet scoring into `scripts/systems/battle/packet_scorer.gd` as a single pure function `static func score(chapter1: Dictionary, frames: Dictionary, evidence: Dictionary) -> Dictionary`. Both `battle_controller.gd` and `blue_binder.gd` call it. Add a unit test `test_packet_scorer.gd` that fixtures the two callsites' historical inputs and asserts identical outputs. Delete the duplicated methods.
**Severity:** S1

### F5 — Four signals declared and emitted, zero production consumers
**Flaw:** `case_folder_acquired` (signals.gd:16, emitted by `blue_folder_pickup.gd:72`), `case_folder_toggled` (signals.gd:21, emitted by `case_folder.gd:272`), `case_folder_fragment_added` (signals.gd:12, emitted by `dialogue_runner.gd:235`), and `judge_skepticism_raised` (signals.gd:136, emitted by `battle_controller.gd:938`) are each emitted in production but `connect`-ed only from `tests/`. `coffee_brewing_completed` is in the same category but is at least documented as redundant-with-`minigame_finished`.
**Why this fails:** A signal with no production receiver is either a missing consumer or a dead signal. Either way, future contributors cannot tell which without reading every file. Encourages cargo-culting `signal X; X.emit()` patterns.
**Remediation:** Pick a path per signal. If a consumer is genuinely deferred (badge popup, route-unlock toast, judge-skepticism camera shake), file the consumer ticket and add a `## TODO(consumer): <ticket-id>` line on the signal declaration. If the signal has no planned consumer, delete the declaration and the emit. Add a CI check (`grep`) that fails when a non-TODO signal is declared in signals.gd with no production `.connect` outside `tests/`.
**Severity:** S2

### F6 — `BinderUI` autoload undocumented; canonical list out of date
**Flaw:** `godot/project.godot:28` registers `BinderUI="*res://scripts/autoload/binder_ui.gd"`. `AGENTS.md:62` and root `CLAUDE.md` Project Snapshot both say "Autoloads: State, Signals, Casebook, DialogueRunner" — four, not five. No SPRINT_LOG entry I can find updates the canonical list.
**Why this fails:** The autoload list is the load-bearing "what is global" claim. Drift here means future PRs can pile more singletons in without anyone noticing the discrepancy. Also makes `--script` test mode subtly wrong: tests that assume the four-autoload world will not see BinderUI unless they explicitly load `scripts/autoload/binder_ui.gd`.
**Remediation:** Update `godot/AGENTS.md`, root `AGENTS.md`, root `CLAUDE.md`, and `godot/CONVENTIONS.md` to list BinderUI as the fifth autoload, with a one-line description of its scope. Add a smoke assertion that `OS.has_feature("editor")` aside, `/root/BinderUI` resolves at boot. If BinderUI is judged not to deserve autoload status, demote it to a child of MainController and reach it via `MainController.binder` (the pattern RoomTransition already uses).
**Severity:** S2

### F7 — Door required-flag lookup uses top-level keys, not chapter1 paths
**Flaw:** `scripts/actors/door.gd:91` checks `st.get(required_flag, false)`. `st` is `State.data`. Chapter 1 flags live at `State.data.chapter1.<flag>` — `has_law_binder`, `has_case_folder`, `won_court`, etc. A door declaring `required_flag: "has_law_binder"` in doors.json reads `State.data.has_law_binder` (missing → false), so the door is permanently locked. Today this works only because every entry in `doors.json` has `required_flag: ""`. The first chapter1-gated door silently fails.
**Why this fails:** Two flag-resolution conventions in one codebase. DialogueRunner walks dotted paths via `_resolve_path`; CaseFolderModel walks dotted paths via `resolve_path`; Door walks one level. A new contributor will not predict which works where.
**Remediation:** Replace `st.get(required_flag, false)` with a dotted-path resolver mirroring `DialogueRunner._resolve_path`. Document the expected format in `doors.json._doc` ("dotted path into State.data, e.g. `chapter1.has_law_binder`"). Add a test `test_door_required_flag.gd` that fixtures both a top-level and chapter1 flag and asserts the gate behaves.
**Severity:** S2

### F8 — `door.gd:88` calls `reset_state()` only to discard it
**Flaw:** `var st: Dictionary = state_node.reset_state() if state_node else {}` — allocates a fresh default state dictionary every time a door is opened, then overwrites it with the actual `state_node.data` on the next line. The branch reachable only when `state_node` is null is dead because line 89 also checks `state_node`.
**Why this fails:** Allocates ~3KB of dictionary churn per door-open, with no functional purpose. The branching is cognitively misleading — a reader has to think about when the default-state path matters; it never does.
**Remediation:** Replace with `var st: Dictionary = state_node.data if state_node else {}`. While editing, fold the duplicate state lookup at `door.gd:87-90` into the dotted-path resolver from F7.
**Severity:** S3

### F9 — `_argument_fragment_definition` reads JSON from disk on every dialogue mutation
**Flaw:** `dialogue_runner.gd:238-252` opens, reads, parses `res://data/argument_fragments.json` every time a dialogue's `on_dismiss` contains `add_argument_fragment`. The autoload already runs at boot and caches `_catalogue`, `_character_registry`, `_known_speaker_ids` — but the argument-fragments file is re-read on every fire. Web export will hit this on every player-driven argument-building dialogue choice.
**Why this fails:** Per-event disk IO inside an autoload is the class of latency spike that surfaces as random hitching late in development. Once the binder-read pattern lands, this fires repeatedly within a single conversation.
**Remediation:** Add `var _argument_fragments: Dictionary = {}` and a `_load_argument_fragments()` called from `_ready()` alongside `_load_character_registry()`. `_argument_fragment_definition(fragment_id)` becomes a dict lookup. Keep the JSON path constant; preserve the existing public surface.
**Severity:** S3

### F10 — `_load_judgments` / `_load_opponents` load GDScript classes inside loops
**Flaw:** `battle_controller.gd:572` and `:590` each call `load("res://scripts/systems/battle/judgment.gd") as GDScript` (and the equivalent for argument_opponent) inside the `for raw_judgment in raw_judgments:` loop. ResourceLoader caches by path, so this is not catastrophic, but the loop allocation is unnecessary and obscures intent — the read-before-write principle (AGENTS.md Rule 8) asks you to think about lifecycle.
**Why this fails:** Same load() called N times where 1 suffices. On web export, ResourceLoader cache hits are still measurable. The pattern also drifts the script reference into per-iteration scope, so a typo in the path fails per-item rather than at boot.
**Remediation:** Hoist `const JUDGMENT_SCRIPT := preload("res://scripts/systems/battle/judgment.gd")` and `const OPPONENT_SCRIPT := preload("res://scripts/systems/battle/argument_opponent.gd")` to file scope. Loop bodies become `JUDGMENT_SCRIPT.load_from_dict(raw_judgment)`. Same edit in `_load_opponents`.
**Severity:** S3

### F11 — `dialogue_runner.gd` re-asserts mutations queue twice per match
**Flaw:** Lines 640 and 678 both execute `_ctx.mutations = raw_mutations.duplicate(true)`. The first runs before emitting `dialogue_line_ready`; the second runs after, with the rationale (lines 674-677) that "Signal handlers may run synchronously while a chained state is being loaded". This is defensive code for a real bug class, but the duplicate deep-copy on every state match is per-event allocation that can be avoided by establishing the invariant in one place.
**Why this fails:** Two assignments to the same field within ten lines is the kind of redundancy that gets pruned by a "cleanup" pass two sessions from now — and then the chain re-fire breaks subtly. The fix is to make the invariant testable, not to rely on prose.
**Remediation:** Encapsulate the "the matched state's dismiss queue must outlive synchronous signal handlers" rule into a single helper, e.g. `_ctx.install_mutations(raw_mutations)` that the chain re-fire path explicitly re-invokes. Then write a focused test (`test_dialogue_chain_mutations_outlive_handler.gd`) that proves the invariant by registering a synchronous `dialogue_line_ready` listener that calls back into the runner. Without that test, the two-line "magic" is unprotected against well-meaning refactors.
**Severity:** S3

### F12 — `dialogue_box.gd` calls underscored methods on the DialogueRunner autoload
**Flaw:** `dialogue_box.gd:234` and `:273` reach into the DialogueRunner autoload to call `_resolve_speaker(character_id, fallback)`. The leading underscore is GDScript's "private by convention" marker; the method's docstring (`dialogue_runner.gd:396-400`) does not mark it as a public API.
**Why this fails:** UI binding to a private autoload method couples renderer to the runner's internal speaker-resolution lifecycle. A refactor of the registry-loading path will silently break the dialogue box. Reach-in patterns also defeat the goal of the Signals bus.
**Remediation:** Add `func resolve_speaker(character_id: String, fallback: String) -> String` as the public surface in DialogueRunner; keep `_resolve_speaker` as the internal implementation that the public method delegates to. Alternatively, expose the character registry shape through `Signals.character_registry_loaded(registry: Dictionary)` so DialogueBox builds its own lookup at boot and never round-trips through the runner at line-render time.
**Severity:** S3

### F13 — `.bak.2026-05-17` files live in active runtime data paths
**Flaw:** `godot/data/dialogues/asia.json.bak.2026-05-17`, `barista.json.bak.2026-05-17`, `cula.json.bak.2026-05-17`, `judge_district_ch1.json.bak.2026-05-17`, `asia_hint_states_ch1.json.bak.2026-05-17`, plus `godot/data/character_registry.json.bak.2026-05-17`, `godot/data/asia_hints.json.bak`, and `godot/data/dialogues/halina.json.bak`. The runtime catalogue loader at `dialogue_runner.gd:413` skips them by `.ends_with(".json")`, so they don't pollute the catalogue — but they sit inside the same directories that the dialogue editor and voice audit walk. Memory entry [scene .tscn files uncommitted] already names this hygiene gap; data/ shows the same pattern in a directory that AGENTS.md calls "runtime JSON data".
**Why this fails:** Stale `.bak.<date>` files in a runtime directory are how a tooling script picks up a stale schema and a contributor wastes an afternoon debugging "why is asia.json not behaving like the file I just edited?" — the editor opened the .bak. AGENTS.md operating-principle "Surface conflicts, don't average them" applies here.
**Remediation:** Move all `.bak.<date>` files under `godot/data/_drafts/` or a new `godot/data/_archive/<date>/` directory. Add `*.bak`, `*.bak.*` to `.gitignore` for `godot/data/**` so future autosaves don't ship as commits. Audit `tools/voice_audit.py` and `tools/verify_dialogue_roundtrip.js` to confirm they explicitly filter by `.json` extension (voice audit already does; verify_dialogue_roundtrip needs checking).
**Severity:** S3

## Standing concerns

- BUILD_NOTES 2026-05-16 records that the test runner returned green for 45 sessions while running zero tests. The codebase has corrected the symptom; the underlying class — "an aggregator that defaults to success" — is worth a standing invariant check in F1's smoke-error pass.
- The `chapter1` flag-bag is now 60+ keys flat in `State.data.chapter1`. Every new beat adds 5-10 booleans; `state.gd` is up to 295 lines and growing linearly with content. Worth tracking whether at chapter 2 the right move is a per-chapter sub-dict (`State.data.chapter2.<flag>`) or a generalised flag store with namespaces. Either decision should land before SAVE_VERSION 25, not after.
- The `Dictionary[String, float]` typed-dict signatures in `battle_controller.gd` are correct GDScript 4.x but rare across the codebase — most other autoloads use plain `Dictionary`. Lock down which convention applies in `CONVENTIONS.md` so the pattern doesn't fork.
