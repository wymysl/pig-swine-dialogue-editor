# Game & Systems Design — Hostile Critique, 2026-05-26

## Verdict
The Casebook Battle System wears a Pokémon UI in the design doc and ships as a flag-driven dialogue tree in the runtime. The Phase 1 → Phase 2 carry-over PROPOSALS.md §10 promised is real for exactly one round of one chapter, and inside that round Phase 2's outcome is computed by `consume_assembled_packet` before the player invokes a single citation. The taxonomy has two articles; the Casebook has one judgment; the live battle UI has not surfaced. Mechanical depth has been spent on flag plumbing and proposal documents, not on decisions the player can see or feel.

## Findings

### F1 — One judgment, one opponent, no matchups
**Flaw:** `data/judgments.json` contains a single judgment (`procedural_reset_ch1`) with five moves that share the same Article + Principle + Context tag set. `data/argument_opponents.json` contains a single opponent (`landlord_counsel_ch1`). `data/tag_taxonomy.json` exposes 2 article tags, 10 principle tags, 7 context tags — and the docstring says Ch2+ tags will land only when an encounter consumes them.
**Why this fails:** Type-effectiveness mechanics require types. With one judgment whose moves overlap on tags, the resolver has nothing to resolve — every move lands inside the same article/principle band against the same opponent. The player cannot learn matchups they will never face. The Pokémon analogy that animates `battle_mechanics.txt` requires a Casebook the player picks *from*; the Casebook is a single card.
**Remediation:** Either author two more Ch1 judgments with deliberately misfit tags (an Article 8 home judgment, an Article 10 expression judgment — both wrong-for-Sikorska so the player learns by missing) and ship them as overworld pickups, or admit Ch1 is single-issue and drop the type-effectiveness framing from `battle_mechanics.txt` for chapters 1-2. Touch `data/judgments.json` and `data/tag_taxonomy.json`; align the spec.
**Severity:** S1

### F2 — `consume_assembled_packet` decides the verdict before Phase 2 runs
**Flaw:** `scripts/systems/battle/battle_controller.gd` lines 442–468. On entering any non-Phase-1 round (or at end_round) the controller calls `consume_assembled_packet()`, which derives `proposed_frame`, `halina_trust`, `decoy_overbroad_remedy`, `recruited_crab`, and writes `court_outcome` to one of `OUTCOME_STRONG`/`STANDARD`/`NARROW`/`BLUNDER_RECOVERED` based on flags the player set during dialogue. Phase 2's `player_present` then runs the live tag resolver against the opponent move, but `_round_buckets` is the only state that depends on those plays — `court_outcome` is already written.
**Why this fails:** PROPOSALS.md §10's load-bearing claim is that Phase 1 fact-flags gate Phase 2 citations, *and Phase 2 citations are how you win*. The implementation reduces Phase 2 citations to flavor: the outcome was decided when the packet was assembled in dialogue. The "court round" the player thinks they're playing is a curtain over a resolved verdict.
**Remediation:** Remove `court_outcome` write from `consume_assembled_packet` (line 465). Compute `court_outcome` at `end_round` (after round 3) from the live record of Phase 2 citation buckets plus packet completeness, not packet completeness alone. Decide which layer is decisive — packet assembly OR Phase 2 citation play — and cut the other to flavor. Currently both layers exist but only one is heard.
**Severity:** S1

### F3 — Phase 1/Phase 2 carry-over exists for round 1 only; rounds 2 and 3 are paste
**Flaw:** `data/court_rounds/` contains `chapter1_round_1.json` (1,037 lines, full fact-flag wiring) and `chapter2_round_1.json` (draft stub). There is no `chapter1_round_2.json` or `chapter1_round_3.json`. The controller drives rounds 2 and 3 from `argument_opponents.json::landlord_counsel_ch1.court_rounds[]` — opponent move definitions with no per-round fact-flag plumbing. `start_round` recurses `start_round(opponent.id, round_index + 1)` at `end_round`, so rounds 2 and 3 inherit the round-1 phase-one cooperation budget reset and never authored their own Phase 2 frame gates.
**Why this fails:** The two-phase split was sold as the chapter-structuring move. As shipped, Round 1 has the structure and Rounds 2-3 collapse back to "pick a Press/Present, get a bucket, advance." The remedy-discipline rule that's supposed to gate Round 3 (story.txt Beat 12 "remedy discipline (load-bearing)") lives in `judgments.json::procedural_reset_ch1.remedy_discipline` as a description string, not as a Round 3 mechanic.
**Remediation:** Author `chapter1_round_2.json` and `chapter1_round_3.json` to the same schema as Round 1, or restructure: declare Ch1 has one true two-phase round and Rounds 2 and 3 are post-verdict scenes (closing arguments, remedy ask) that resolve without their own fact-finding. Either move forces the design honest. The current shape is the worst of both: a recursive `start_round` over data files that do not exist.
**Severity:** S1

### F4 — The battle has no battle screen
**Flaw:** Per the data layer and `casebook_judge_state` flag transitions written by `battle_controller.gd`, the court rounds drive `data/dialogues/judge_district_ch1.json` and `data/judge_reactions_ch1.json` via dialogue states. The "battle UI" the spec calls for in `battle_mechanics.txt` (opponent portraits, Argument Strength meters, effectiveness popups, judgment buttons) is not in the runtime — the IMPLEMENTATION_mechanical_depth_2026-05-18.md §A Trial Record panel was the proposed minimum visibility layer and remains undelivered.
**Why this fails:** The legibility of Pokémon comes from the UI exposing the system — type icons, the "It's super effective!" popup, HP bars. With the system hidden behind dialogue text, the player has no surface to learn the grammar from. The 1,037-line tag-weighted Phase 1 file and the dot-product effectiveness resolver run invisible to the player. Per the player's eyes the game is dialogue with consequences; the Casebook system might as well not exist.
**Remediation:** Ship the Trial Record panel (IMPL §A) as the minimum legibility surface for Phase 2 *before* any further mechanical authoring. Concretely: `scripts/ui/trial_record_panel.gd`, `data/court_rounds/_trial_record_labels.json`. Then revisit whether the tag resolver is doing player-visible work or whether it should be cut as overengineering of a flag-gate system.
**Severity:** S1

### F5 — Bucket spread collapses against the single Ch1 opponent
**Flaw:** `data/argument_opponents.json::landlord_counsel_ch1.court_rounds[0].moves[0].immune_to: []`, `resists: ["proportionality", "margin_of_appreciation"]`. The Ch1 weighted move tags carry `service_of_process`, `procedural_fairness`, `access_to_court` weighted ~0.2 each. `effectiveness.gd::BUCKET_THRESHOLDS` requires score ≥ 0.70 for super_effective. Tag overlap between the procedural-reset moves and the opponent's `weak_to` set (`service_of_process`, `access_to_court`, `procedural_fairness`) means almost every well-formed Ch1 move resolves to "super_effective" or "effective" — the lower buckets are practically unreachable because the player cannot field a `proportionality`-tagged move (no judgment carries it).
**Why this fails:** Five buckets need scenarios that span them. In Ch1 the player has two buckets in practice. The resolver is doing math nobody can lose. "Backfires" requires the player's primary tag to land on opponent strength tags; opponent strength tag dictionaries are short or absent. The bucket system is overengineering of what `requires_fact_flags` (binary gate) already does for free at the citation level.
**Remediation:** Either populate `immune_to`/`resists` on opponent moves so wrong-tag picks fail loudly, plus author Ch1 judgments that *can* fail (see F1), or drop the bucket resolver for Ch1 and use binary citation availability gating — which is what `chapter1_round_1.json` already does for Phase 2's `requires_fact_flags`. Do not run both systems in parallel.
**Severity:** S2

### F6 — `battle_mechanics.txt` is 2,751 lines of obsolete design
**Flaw:** The spec opens with Wild Argument encounter design, sets up "Casebook collection" as a player goal, recommends "P0 should use 10 judgments maximum," prescribes Training Battles, and walks through encounter rates. PROPOSALS.md §1 (status DONE 2026-05-04) cut wild encounters, deferred collection, and reframed Final Printer. PLAN.md §Out of scope permanently confirms the cuts. AGENTS.md §Forbidden patterns explicitly forbids "Building wild-argument overworld encounters."
**Why this fails:** AGENTS.md §"Source of Truth": *"Creative canon: the five root `.txt` files."* The canonical battle spec contradicts the active design. Any new agent reading the spec top to bottom — which AGENTS.md §"Reading order on every invocation" mandates — will build the wrong game. The contradiction is twelve months old.
**Remediation:** Rewrite `battle_mechanics.txt` against the actual P0: one judgment per chapter, scripted court rounds, two-phase Press/Present, packet assembly, no Wild Arguments. Strip §"Encounter types" sub-1, 2, 4. Strip §"Casebook progression" unlock methods that reference wild encounters and training. This is human-owned per AGENTS.md and has been "DEFERRED" since 2026-05-04. It is now an active hazard.
**Severity:** S2

### F7 — Trust meter is a switch dressed as an integer
**Flaw:** Per the auto-memory and `argument_frames_ch1.json` deltas: trust starts at 0, decoy penalties are -1/-1/-4. Per `data/items.json`, the Beat 8 stance pick gates a single-value `bonus_evidence_collected` enum (Wojcik / return slip / lease). Per memory the reveal threshold is trust ≥ 5. The runtime affords one stance pick + one optional incapacity penalty. There is no path that exercises the integer's range beyond {start, post-incapacity}.
**Why this fails:** A meter implies repeated decisions that accumulate. Halina's runtime is one stance branch + one fault state. Calling it a meter oversells it. The reveal threshold ≥ 5 is reachable only by paths where trust never drops; the integer never lives in interesting territory like 2-4. The meter is a 0/5 switch with a -4 penalty bolted on.
**Remediation:** Either (a) build trust as a sequence: 3-4 trust-affecting choices across Beats 4-10 with thresholds at 2, 4, 6, or (b) rename to `halina_stance_branch` enum + `incapacity_penalty` bool and stop pretending the integer is doing dramatic work. Memory `project_pig_swine_trust_meter.md` conflicts with the runtime as authored — reconcile.
**Severity:** S2

### F8 — Chapter 2 round 1 is a 863-line stub gated on six unbuilt files
**Flaw:** `data/court_rounds/chapter2_round_1.json` line 8: `"_status": "... Gated on Ch1 F1 closure (DESIGN_TODO inventory in chapter1_round_1.json) and on six open data dependencies: argument_frames_ch2.json (NEW FILE), argument_opponents.json::landlord_counsel_ch2 (new entry, three rounds), evidence_ch2.json (NEW FILE — building_rules_extract, eviction_notice, poster_photo), judgments.json::rule_clarity_ch2, tag_taxonomy.json Ch2 extension, state.gd::chapter2.round1_rule_argument_won + chapter2.round1_outcome defaults + SAVE_VERSION bump + migration test."` `draft: true`.
**Why this fails:** PLAN.md §Out of scope: *"Chapter 2-5 scaffolding beyond the JSON file headers"* — until Ch1 ships. AGENTS.md §"Forbidden patterns": *"Building Chapter N+1 content while Chapter N is not yet shippable."* This file violates both. It exists primarily to demonstrate schema; it cannot load, cannot test, cannot ship. Reading the repo gives a false impression of Ch2 progress.
**Remediation:** Move `data/court_rounds/chapter2_round_1.json` to `_drafts/` or `git rm`. Author Ch2 court rounds *after* Ch1 has a playable web export and a playtest signal. AGENTS.md Rule 2 (Simplicity First) and PLAN.md §Out of scope already say this; enforce.
**Severity:** S2

### F9 — Incapacity decoy has no in-round visible consequence
**Flaw:** `packet_scorer.gd:247` returns `OUTCOME_BLUNDER_RECOVERED` for `has_incapacity || burns_round_attempt || supported_count <= 1`. `judge_reactions_ch1.json::icy_silence` has the judge pick up the procedural defect himself ("The court will hear no further argument from counsel on the capacity of the respondent. The respondent has spoken for herself. The court accepts her competence on its own observation."). `_packet_recovery_source` always returns a non-empty source. The bench rescues the player. The case is never lost.
**Why this fails:** Soft failure is fine. Bench rescue is the opposite — it tells the player "do anything, the system catches me." The incapacity decoy was sold by `IMPLEMENTATION_mechanical_depth_2026-05-18.md` §C as *"the firm's moral compass made visible."* As wired the cost is a Halina trust drop (silent meter), a Crab withdrawal (silent flag flip, see F10), a Day-One Summary variant, and a Ch4 corridor recolor — all downstream of the round, none felt in the moment. The moral choice has no moral consequence in the room.
**Remediation:** Let the round end on the wrong note. Cula files incapacity, judge declines to engage, bench rules on the procedural defect bench-initiative. *Cula does not speak the winning argument.* Render this as a Cula-internal beat ("I held the wrong argument; the court found the right one without me") that the run summary echoes. Do not let `_packet_recovery_source: court_redirect` resolve invisibly.
**Severity:** S2

### F10 — `consume_assembled_packet` silently flips `recruited_crab` to false
**Flaw:** `battle_controller.gd:463`: `if bool(score.get("crab_support_withdrawn", false)): _write_chapter1_flag("recruited_crab", false)`. Triggered by `packet_scorer.gd:103` when `has_incapacity && recruited_crab`. The flag flip writes through `_write_chapter1_flag`, which emits `chapter1_flag_changed` but has no associated dialogue beat or portrait state.
**Why this fails:** Silent-content-cut failure mode. The player who filed incapacity will later notice Crab's Phase 1 press options have disappeared, with no in-fiction acknowledgment of why. `IMPLEMENTATION_mechanical_depth_2026-05-18.md` §C explicitly required a Crab dialogue line at `chapter1.incapacity_filed == true`; that line is not in `data/dialogues/crab.json` as a triggered state (verify, but the controller's silent flip is the smoking gun). This is the exact failure mode the firm-stability-meter cut (`PROPOSAL_mechanical_depth_2026-05-18.md` §1) was supposed to prevent.
**Remediation:** Block the flag flip on a dialogue beat: emit `signals.crab_withdrew_after_incapacity` from `consume_assembled_packet`, gate `recruited_crab = false` on the dismissal of a Crab state that fires before the next Phase 1 sub-round. Author the state in `crab.json` per IMPL §C. The flag flip without dialogue is content-removal-by-stealth.
**Severity:** S2

### F11 — First moment the system shows itself is Beat 12, after an hour of dialogue
**Flaw:** PLAN.md §Vertical slice plan: Sprint 2 (office + street), Sprint 3 (investigation loop with binder pickup, Crab/Whimsy recruitment, optional coffee), Sprint 4 (Casebook battle system v1), Sprint 5 (court). The first time the player meets the Press/Present surface is the Beat 12 court round. Beats 1-11 are pure dialogue navigation with stance picks but no system-revealing choices. The coffee minigame surfaces a different vocabulary (rhythm timing) that does not preview the court system.
**Why this fails:** A first-time player loaded into the build will spend ~40 minutes reading before the system shows what kind of game this is. The decision space (Press a witness statement, Present an evidence card, watch the judge's tolerance for your move) is the answer to "what kind of game." Hiding it until the end of Ch1 means the player's mental model is set by the dialogue tree, not by the court mechanic. When the court fires they meet a verb they have never used.
**Remediation:** A Beat 3-4 micro-encounter that surfaces the Press/Present verb in a no-stakes Murrow training run, gated only by binder pickup. One witness (Murrow as drill partner), three statements, three press options, one Present. Teaches the system. Authoring shape exists already — clone the round-1 phase_1_fact_finding pattern, downscale to one statement, label it a rehearsal. Resolves §B PROPOSAL_mechanical_depth's "junior signatures" use case as a side effect.
**Severity:** S2

### F12 — "Bonus evidence" is a flavor multiplier on a single stance pick
**Flaw:** `data/items.json::wojcik_witness_statement.stance_gate: "sympathetic"`, `return_to_sender_slip.stance_gate: "blunt_procedural"`, `lease_1962_inheritance_1987.stance_gate: "technical"`. `chapter1.bonus_evidence_collected` is a single-value string enum. The player picks one of three; the other two never appear.
**Why this fails:** "Bonus" suggests stacking or optional accumulation across a run. As wired, the bonus evidence is a tag-pick on the stance decision, full stop. The player cannot deploy both Wojcik's statement and the return slip on the same run. The naming oversells what is a flavor multiplier on a single branch decision.
**Remediation:** Either rename to `client_meeting_evidence` (single-value enum, no "bonus" framing) or drop `stance_gate` and let the player collect multiple cards across the meeting through additional press exchanges. The current shape misrepresents itself in items.json and in critique reports that have to keep explaining it.
**Severity:** S3

## Standing concerns
- Runtime is consistently ahead of the spec. `PROPOSAL_player_driven_argument.md` §11 documents a six-item SPEC_SYNC backlog the runtime is already authoring against. PROPOSAL §10 is marked DONE but the spec text never landed (`PROPOSAL_court_rounds_schema.md` is REVERTED). AGENTS.md promises root specs are authoritative; the runtime ignores that promise to ship. Pick one: either the spec leads, or admit the spec follows and demote it from "Source of Truth" to "frozen reference."
- Proposal-to-implementation ratio is unsustainable. `PROPOSAL_player_driven_argument.md` is at v3, `PROPOSAL_mechanical_depth_2026-05-18.md` has four DEVELOP items with no shipped evidence, `PROPOSAL_court_rounds_schema.md` is REVERTED. For a single-developer hobby project, this is a planning sink. Cap proposals-in-flight at one until §A Trial Record panel ships.
- Chapter 1 has been "shipping vertical slice" since 2026-05-04. PLAN.md called for 14-20 sessions. The current sprint count is well past that. There is no evidence in the repo of a successful `godot --headless --export-release "Web"` against the current battle controller branch. The first acceptance test for the slice — "a stranger can sit down with the web build and play through" — is the unanswered question that should govern every next sprint.
