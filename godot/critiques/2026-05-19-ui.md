# UI & UX — Hostile Critique, 2026-05-19

## Verdict
The Case Folder ships its `_doc: DRAFT` placeholder strings to the player; the Motion Packet is mouse-only because every `OptionButton`, `Button`, and `CheckBox` in `blue_binder.gd` explicitly sets `focus_mode = Control.FOCUS_NONE`; save failure prints `push_error` and shows the player nothing. The build is not playable on keyboard for the chapter-1 climax and is actively lying to the player about the rest.

## Findings

### F1 — Stop shipping `_doc: DRAFT` strings as live UI text
**Flaw:** `godot/data/case_folder_strings.json` contains the live value `"folder_title": "_doc: DRAFT - CASE FOLDER - DR. A. CULA"`, with every `tab_labels`, `empty_states`, `detail_labels`, and `new_badge_label` field carrying the same `_doc: DRAFT - ` prefix. `case_folder.gd::_build_tabs` and `case_folder_model.gd::tab_label` read these verbatim and feed them straight to `Button.text` and `Label.text`. The Title Strip in `scenes/ui/case_folder.tscn` renders the literal string `_doc: DRAFT - CASE FOLDER - DR. A. CULA` to the player on every open.
**Why this fails:** Discoverability and trust. The Case Folder is the chapter-1 evidence surface; the first thing the player sees on B is internal scaffolding leaking through the chrome.
**Remediation:** Replace every `_doc: DRAFT - …` value in `data/case_folder_strings.json` with the final string. Add `tests/test_case_folder_strings.gd` that asserts no value in that JSON contains the substring `_doc:` or `DRAFT`.
**Severity:** S1

### F2 — Motion Packet is mouse-only by design
**Flaw:** `scripts/ui/blue_binder.gd` sets `focus_mode = Control.FOCUS_NONE` on every `OptionButton` (line 161 of `blue_binder.tscn`, etc.), the `ApplyPacketButton`, all tab buttons (`_make_tab` line 375), and the decoy `CheckBox` nodes (`_build_decoy_options` line 540). `_unhandled_input` only handles `ui_left` / `ui_right` / `ui_cancel`. There is no keyboard handler for slot assignment, remedy choice, decoy toggling, or "Apply Packet Assessment."
**Why this fails:** Input parity. The chapter-1 court round is gated on submitting a packet, and the packet cannot be assembled without a mouse. Steam Deck and trackpad-only laptop sessions cannot complete the chapter.
**Remediation:** Drop `focus_mode = Control.FOCUS_NONE` everywhere in `blue_binder.gd` and `case_folder.gd`. Add a keyboard focus traversal: Tab / Shift-Tab through slots, Up/Down to change `OptionButton` selection without opening the popup, Enter on `ApplyPacketButton`. Write `tests/test_blue_binder_keyboard_nav.gd` that simulates a full keyboard-only packet submission.
**Severity:** S1

### F3 — Save failure shows the player nothing
**Flaw:** `scripts/systems/save.gd` handles every failure path (`State` missing, can't create user dir, can't open for writing, JSON parse failed) with `push_error` and a silent `state.reset_state()`. There is no toast, no modal, no log line surfaced to the UI. On web export, a quota-exceeded or corrupted `user://save.json` deletes hours of progress with zero feedback.
**Why this fails:** Error states. The player thinks they saved; the next session boots into a fresh chapter-1 office. Indistinguishable from "game forgot me."
**Remediation:** Add `Signals.save_failed(reason: String)` and `Signals.save_completed()`. Wire a small `scenes/ui/save_status_toast.tscn` (a `CanvasLayer` Label that fades over 3s) into `Main.tscn`. Replace each `push_error` in `save.gd` with both the existing log line and a signal emit. Add `tests/test_save_failure_signal.gd` that forces a write to a read-only path and asserts the signal fires.
**Severity:** S1

### F4 — Interaction prompt hardcodes `[E]`
**Flaw:** `scenes/ui/interaction_prompt.tscn` sets `Label.text = "[E]"` directly in the scene. There is no rebind hook. If the player rebinds `interact` to F or a controller button, the prompt above every NPC still reads `[E]`.
**Why this fails:** Consistency and accessibility. WCAG and basic rebinding hygiene both require the prompt to reflect the current binding.
**Remediation:** In `interaction_prompt.gd::_ready`, query `InputMap.action_get_events("interact")` and write the resolved human label into the `Label`. Surface a `set_action_name(action: String)` method so the same prompt scene can serve `case_folder_toggle` and `sprint` later.
**Severity:** S2

### F5 — No text-speed setting; typewriter is hardcoded
**Flaw:** `scripts/ui/dialogue_box.gd` declares `const TYPE_SPEED_CHARS_PER_SEC: float = 90.0` and `const TICK_INTERVAL: float = 0.06`, with no exposed setter or config path. There is no "instant text" mode, no per-player speed override, no skip-on-second-pass.
**Why this fails:** Accessibility. Slow readers, screen-reader users, and replayers all need a speed control. Chapter 1 contains long Halina testimony and Pig panic monologues; locking pace to 90 cps is a hostile default for any non-typical reader.
**Remediation:** Add `State.data.settings.text_speed_chars_per_sec: float` (default 90), a settings menu surface (deferred can be a hidden debug overlay first), and read it in `dialogue_box.gd::_process`. Add an "Instant" option that sets `_text_label.visible_characters = -1` immediately on `_show_page`. Add `tests/test_dialogue_text_speed_setting.gd`.
**Severity:** S2

### F6 — Cooperation and Patience bars carry no text
**Flaw:** `scenes/ui/battle_screen.tscn` defines `CooperationBar` and `PatienceBar` as bare `ProgressBar` nodes (`max_value = 10.0`, `value = 10.0`) with no companion `Label` and no `show_percentage` override. State is encoded as fill width only.
**Why this fails:** Accessibility (no information by color or fill-bar alone — explicitly forbidden by `AGENTS.md` §Accessibility), and discoverability — the player cannot tell `7/10` from `8/10` at a glance.
**Remediation:** Wrap each `ProgressBar` in an `HBoxContainer` with a numeric `Label` showing `value/max_value`, updated by the controller. Add a contrast-checked icon next to each bar (gavel for patience, handshake for cooperation) so the meaning survives even when text scaling is on.
**Severity:** S2

### F7 — `scenes/ui/battle_screen.tscn` is wired to a dead script
**Flaw:** `scripts/ui/battle_screen.gd` is two lines: a comment saying "REVERTED — premature skeleton" and an `extends Node`. The scene still exists with `PhaseLabel`, `JudgeSpeechBox`, `WitnessSpeechBox`, `CooperationBar`, `PatienceBar`, `OptionsContainer`, `ResultOverlay`. If anything in `Main.tscn` or a transition path instantiates this scene, the UI renders with no input handling.
**Why this fails:** Feedback latency and error states. Dead scenes hide behind the export — the next agent who connects the controller will find these nodes positioned with anchor presets `-1` (custom), all overlapping in the top half of the viewport with no layout discipline.
**Remediation:** Either delete `scenes/ui/battle_screen.tscn` and the `.gd` stub together, or rewrite the script to a real controller this sprint. If keeping the scene, add `tests/test_battle_screen_skeleton_not_instantiable.gd` that fails if anything outside `tests/` references this path.
**Severity:** S2

### F8 — `case_folder_toggle` has no in-game discovery
**Flaw:** `project.godot` binds `B` to `case_folder_toggle`, but nothing on screen tells the player. The Asia hint surface mentions the binder only in dialogue; there is no HUD glyph, no first-time-acquired toast when `chapter1.has_case_folder` flips true. `scripts/ui/case_folder.gd::_can_open` silently no-ops if the flag isn't set.
**Why this fails:** Discoverability. The first time the binder is needed (Beat 7 onward) the player has to remember a key the game never showed them. A no-op silent rejection on `B` before the flag flips is worse — it teaches the player the key does nothing.
**Remediation:** When `chapter1.has_case_folder` flips true, push a one-shot toast via `Signals.case_folder_acquired` reading "Press [B] to open your case folder" — with the same rebind-aware lookup as F4. Add a persistent HUD glyph in the corner (small folder icon + bound key). Add `tests/test_case_folder_toast_fires_once.gd`.
**Severity:** S2

### F9 — `binder` input action is bound to no events
**Flaw:** `project.godot` declares `binder={"deadzone": 0.5, "events": []}`. The action exists, has no key, and is referenced nowhere productive. It is either a renamed-but-not-cleaned-up legacy of `case_folder_toggle`, or a future binding that someone forgot.
**Why this fails:** Consistency. Dead actions drift into rebind menus and confuse the next sprint.
**Remediation:** Decide now: either bind it (and use it) or delete the entry from `project.godot`. Run `rg "\"binder\"" godot/` and resolve every reference.
**Severity:** S3

### F10 — InteractionPrompt label is 12px on a 1280×720 viewport
**Flaw:** `scenes/ui/interaction_prompt.tscn` sets `theme_override_font_sizes/font_size = 12` on the `[E]` label, inside a 28×20px `ColorRect`. At the project's target viewport, that is roughly 9pt physical on a 13" MacBook display.
**Why this fails:** Accessibility — sub-readable for low-vision players and on Steam Deck's small screen. `AGENTS.md` mandates WCAG AA for UI text; 12px body weight does not meet that bar for an action prompt.
**Remediation:** Bump to 16 (matching the prompt's role in pixel-art register), increase the `ColorRect` size to fit (`offset_left/right = -18/18`, `offset_top/bottom = -12/12`), and re-verify y-sort offset against the new height in `interaction_prompt.gd::PROMPT_HEIGHT_OFFSET`.
**Severity:** S3

### F11 — BlueBinder FooterHint lies about what's bound
**Flaw:** `scenes/ui/blue_binder.tscn` line 228 renders `"[Tab/← →] evidence    [Esc] close"`. In reality `blue_binder.gd::_unhandled_input` only handles `ui_left`, `ui_right`, `ui_cancel` — Tab does nothing because every tab Button has `focus_mode = Control.FOCUS_NONE` (see F2). There is also no documented binding for the slot OptionButtons, the remedy, the decoy checks, or Apply.
**Why this fails:** Consistency and feedback latency. The footer is a contract; failing the contract is worse than not making it.
**Remediation:** Once F2 is fixed and Tab actually cycles tabs, the hint becomes true. Until then, change the string to `"[← →] evidence  [Esc] close  (mouse required for packet)"` — or do not ship the binder.
**Severity:** S2

### F12 — Tree-pause modals have no web-export focus-loss strategy
**Flaw:** Both `dialogue_box.gd::_on_dialogue_line_ready` and `case_folder.gd::open` set `get_tree().paused = true`. The web export inherits this on every modal. There is no `notification(NOTIFICATION_WM_WINDOW_FOCUS_OUT)` handler anywhere in `scripts/ui/`. If the browser tab loses focus during dialogue, the player returns to a paused tree with no on-screen indication that input has stopped flowing.
**Why this fails:** Web-export ergonomics. Browser focus loss is the single most common interruption on web; the game becomes "is it frozen?" without one.
**Remediation:** In `dialogue_box.gd` and `case_folder.gd`, handle `NOTIFICATION_APPLICATION_FOCUS_OUT` / `NOTIFICATION_APPLICATION_FOCUS_IN`. On focus-out from a paused modal, show a `"Click to resume"` overlay; on focus-in, hide it. Add an autosave on focus-out for any non-modal scene. Cover with `tests/test_web_focus_modal_overlay.gd`.
**Severity:** S2

### F13 — Manual save UI is specified but not built
**Flaw:** `AGENTS.md` §Stack invariants states "one manual save from the case bag." `scripts/ui/case_folder.gd` exposes four tabs (Notes, Evidence, Casebook, Motion Packet) and never calls `save_game()`. `scripts/systems/save.gd::save_game` exists but no UI surface invokes it outside of autosave hooks. The player has no manual save action.
**Why this fails:** Consistency and error states. A player who suspects a save bug has no way to force a save; a player at a save-worthy moment has no commit button.
**Remediation:** Add a fifth bottom-row "Save Now" button to `case_folder.tscn`, wire it to `save.save_game()`, surface success/failure via F3's `save_failed` / `save_completed` signals into the existing `EmptyState` label area. Add `tests/test_case_folder_manual_save.gd`.
**Severity:** S2

### F14 — Speaker label and dialogue text use near-identical font sizes
**Flaw:** `scenes/ui/dialogue_box.tscn`: `SpeakerLabel` is `font_size = 21`; `TextLabel` is `font_size = 20`. The hierarchy difference is one pixel.
**Why this fails:** Information hierarchy. The speaker attribution is supposed to read as a header; at 21/20 it reads as part of the body, especially when the player is scanning quickly through chained lines.
**Remediation:** Set `SpeakerLabel` to 24 with a bold theme variation; keep `TextLabel` at 20. Verify by running the existing `test_dialogue_typewriter.gd` and adding a layout-snapshot check in `tests/test_dialogue_box_layout.gd`.
**Severity:** S3

## Standing concerns
Three patterns recur across this audit. First, `focus_mode = Control.FOCUS_NONE` is applied as a default in every new UI scene — it removes focus rings, kills tab order, and forecloses keyboard parity before the first sprint that needs it. Second, placeholder content (the `_doc: DRAFT` strings, the dead `battle_screen.gd`, the empty `binder` action) ships in `data/` and `project.godot` with no test gating their removal — every "we'll polish it later" lands in front of the player. Third, the web export target is named in `AGENTS.md` but no scene handles browser-specific concerns (focus loss, storage quota, autosave-on-blur); the build acts like a desktop game in a tab.
