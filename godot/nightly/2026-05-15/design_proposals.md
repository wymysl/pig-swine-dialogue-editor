## Proposal 1: Author Beat 9 — Murrow Archive Room Narration States

**Type:** DIALOGUE  
**Addresses:** story.txt §Beat 9; chapter1.json flag note "not yet authored";
V1.4 phase-7 pack (`narrative_revision/phase_7_packs/V1.4_halina_meeting_research.md`)  
**PROPOSALS.md status:** not mentioned as a standalone entry (pre-work implied by §10 Status)

### Gap

`chapter1.archive_research_complete` is declared in `chapter1.json` with an
explicit note — *"not yet authored; flag declared for V1.A state 8 dispatch"* —
and `chapter1.article_135bis_understood` is referenced in `story.txt` §Beat 12
as gating Cula's strongest court arguments but is not yet declared anywhere in
`chapter1.json`. `murrow.json` has no states that fire in the Beat 9 window
(`met_murrow == true && halina_met == true && !archive_research_complete`).
This is a blocking path: without `archive_research_complete`, Whimsy's
`after_recruitment_court_ready` state never fires, Crab's `crab_hint_court_ready`
never fires, and Murrow's own `court_readiness_check` never fires. Beat 10 is
unreachable from Beat 8 through normal gameplay.

A companion nightly draft is provided at
`godot/data/_drafts/nightly_design_murrow_beat9_2026-05-15.json`.

### Proposed addition

**File to edit:** `godot/data/dialogues/murrow.json`

Add four states after the existing `court_readiness_check` state:

1. **`murrow_archive_narration`** — trigger:
   `chapter1.met_murrow == true && chapter1.halina_met == true && !chapter1.archive_research_complete`  
   Content: Murrow narrates Article 135-bis § 2 KPC clause-by-clause (11–13
   lines), applying each clause to Halina's specific facts. Ends with an options
   block (`write_path: "chapter1.murrow_beat9_dwell"`, `chain: true`) presenting
   Cula's three dwell choices.

2. **`murrow_archive_dwell_archival`** — trigger:
   `chapter1.murrow_beat9_dwell == 'archival_aside'`  
   One or two archival-aside lines. Murrow names which shelf the article lives
   on; the brevity and specificity are the joke. `on_dismiss`: set
   `chapter1.archive_research_complete = true`,
   `chapter1.article_135bis_understood = true`.

3. **`murrow_archive_dwell_prior`** — trigger:
   `chapter1.murrow_beat9_dwell == 'prior_case'`  
   Two lines. Murrow confirms one prior use; the motion failed on timeliness.
   Cula asks about the client; Murrow answers in two words. `on_dismiss`: same
   flags.

4. **`murrow_archive_dwell_next`** — trigger:
   `chapter1.murrow_beat9_dwell == 'next_steps'`  
   One line. Murrow gives the rest of the readiness checklist (Crab, Whimsy,
   filing window). This state bridges cleanly to Beat 10. `on_dismiss`: same
   flags.

**Code dependency (Design cannot complete unilaterally):**  
Two new flags must be declared in `state.gd` `reset_state()` and wired into a
`save.gd` migration before these states are playable:

- `chapter1.murrow_beat9_dwell` — string, default `""`,
  enum `["archival_aside", "prior_case", "next_steps"]`
- `chapter1.article_135bis_understood` — bool, default `false`

Both must also be added to `chapter1.json`'s `new_state_flags` array.

**Voice constraints (from V1.4 pack §B.5):**  
Murrow uses bare `"Cula"` throughout (post-befriending private register).
He cites `"Article 135-bis, section two of the Code of Civil Procedure"` on
first reference; `"Article 135-bis, section two"` alone after that. No
`KPC` abbreviation in dialogue. Three to four sentences per clause; declarative;
no adjectives; no pep. Archival-aside flickers permitted (faint amusement at
competence, not at the client's situation).

### Acceptance criteria

- `python -m json.tool murrow.json` passes.
- Every `chapter1.*` flag referenced in triggers exists in `chapter1.json`
  `new_state_flags` or confirmed present in `state.gd` via `grep`.
- The three dwell states cannot all fire simultaneously (mutual exclusion via
  `murrow_beat9_dwell == '<value>'`).
- `on_dismiss` on all three dwell states sets both
  `archive_research_complete = true` and `article_135bis_understood = true`.
- A walkthrough with `client_meeting_stance = 'sympathetic'` fires
  `murrow_archive_narration`, resolves via one dwell branch, then on next
  interaction fires `court_readiness_check` (the existing Beat 10 state).
- All lines pass the Taste Standard (5/5); Murrow's clause narration reads as
  a person applying a rule to facts, not as a textbook recitation.

## Proposal 3: Author `data/court_rounds/_schema.md` and Resolve PROPOSALS.md §10 Status Discrepancy

**Type:** MECHANICAL  
**Addresses:** PROPOSALS.md §10 (status DONE vs. schema reverted); PLAN.md
§Vertical slice plan step 4 pre-work  
**PROPOSALS.md status:** §10 — currently shows **DONE**, but the physical
`data/court_rounds/_schema.md` file says "REVERTED — premature schema" and is
otherwise empty. The two are contradictory.

### Gap

`data/court_rounds/_schema.md` reads:

```
# REVERTED — premature schema.
Restored from commit c83feaa when PROPOSALS.md §10 is approved.
`git rm` this file (and `rmdir godot/data/court_rounds/`) on Piotr's machine.
```

This means no court round data schema exists on disk, yet PROPOSALS.md §10
says DONE and cites `data/court_rounds/_schema.md` as completed pre-work. The
contradiction creates a hard blocker: PLAN.md §Vertical slice plan step 4
(Casebook Battle System v1) requires `battle_controller.gd` to implement both
sub-controllers from day one, but Code cannot spec the sub-controllers without
knowing the Phase 1 / Phase 2 data shape. PROPOSALS.md §10 specifies the shape
in prose; it needs to be a one-page authoritative schema file so Code can
reference it.

Additionally, `chapter1.json` has no declaration for
`chapter1.witness_cooperation` (the Phase 1 resource counter introduced by §10)
and no declaration for the fact-flags set by Phase 1 outcomes that gate Phase 2
citations. These missing declarations block any Code work on the battle
controller.

### Proposed addition

**File to write (new):** `data/court_rounds/_schema.md`

Content to encode (all material sourced from PROPOSALS.md §10 — no invention):

**Phase 1 block schema** per court round JSON file
(`data/court_rounds/<chapter>_<round>.json`):

```json
{
  "chapter": 1,
  "round": 1,
  "phase_1": {
    "witnesses": [
      {
        "witness_id": "string",
        "witness_cooperation_budget": "int",
        "statements": [
          {
            "id": "string",
            "text": "string",
            "options": [
              {
                "text": "string",
                "type": "press | question | evidence",
                "fact_flag": "string | null",
                "cooperation_delta": "int"
              }
            ]
          }
        ]
      }
    ],
    "fact_flags_available": ["string"]
  },
  "phase_2": {
    "judge_counter_questions": [
      {
        "id": "string",
        "text": "string",
        "available_citations": [
          {
            "judgment_id": "string",
            "requires_fact_flags": ["string"],
            "judicial_patience_delta": "int"
          }
        ]
      }
    ],
    "judicial_patience_budget": "int",
    "weak_win_threshold": "int",
    "strong_win_threshold": "int"
  }
}
```

Include a one-paragraph note per section explaining the Phase 1 → Phase 2
carry-over: fact-flags set in Phase 1 gate which `available_citations` are
usable in Phase 2. A citation whose `requires_fact_flags` list contains a flag
not set in Phase 1 is greyed out in the Casebook UI (shown but not selectable).
This is the load-bearing design mechanic from PROPOSALS.md §10: a sloppy Phase 1
costs the player usable citations in Phase 2.

**File to edit:** `godot/PROPOSALS.md` status table row 10  
Change status from `DONE` to `PENDING (schema on disk, Code pass required)`.
This accurately reflects the current state: the design decision is approved
and the schema is being written, but Code has not yet implemented the
sub-controllers.

**Code dependency:** Once the schema is on disk, Code must:
- Declare `chapter1.witness_cooperation` in `state.gd` (int, default 0)
- Declare Phase 1 fact-flags for Ch1 Round 1 in `chapter1.json`
  (e.g., `chapter1.service_defect_established`, `chapter1.knowledge_imputed`)
- Implement both sub-controllers in `battle_controller.gd` per the schema

**Design dependency:** Once the schema is on disk, Design can author
`data/court_rounds/ch1_round1.json`, `ch1_round2.json`, `ch1_round3.json`
using the V1.6 phase-7 pack as the content source.

### Acceptance criteria

- `data/court_rounds/_schema.md` is non-empty and matches the two-phase shape
  described in PROPOSALS.md §10.
- The Phase 1 and Phase 2 sections are each annotated with a one-paragraph
  explanation of their mechanical role.
- The fact-flag carry-over mechanic (the load-bearing design move from §10) is
  explicitly documented in the schema.
- PROPOSALS.md §10 status no longer claims DONE for a file that does not exist.
- The schema does not invent mechanics beyond what PROPOSALS.md §10 specifies.
