**Type:** MECHANICAL (Code) + DIALOGUE promotion
**Addresses:** `story.txt` §Beat 13 — Return to Pig & Swine — celebration +

## Proposal 2: Wire Office Street prop signage to a JSON data source

**Type:** DATA (Design-owned text + Code-owned interactable wiring)
**Addresses:** `story.txt` §Beat 1 — Office Street arrival; the
`AGENTS.md` §Stack invariants rule that *"all player-facing strings live
in JSON for future i18n. No hardcoded strings in `.gd` or `.tscn`."*
**PROPOSALS.md status:** not mentioned. The 2026-05-18 Pixellab sprints
placed five named-signage Sprite2D props in `office_street.tscn`
(`PigSwineSign`, `NoticeBoard`, `CourtSignpost`,
`StreetNameOfficeStreet`, `StreetNameCounselRow`) but did not wire any
of them to an interactable text source. Today they render as decorative
pixel-art only.

### Gap

`story.txt` Beat 1 says, four times in succession: *"Suggested
objective: (see dialogue_samples.txt §Sign and notice text samples)"*,
*"Example sign text: (see dialogue_samples.txt §Sign and notice text
samples)"*, *"Example notice board text: (see dialogue_samples.txt
§Sign and notice text samples)"*. `dialogue_samples.txt §Sign and
notice text samples` (lines 36–63) holds the source register for these
items — short Asia-adjacent flavor over institutional surfaces.

The five Sprite2D props placed in `office_street.tscn` carry no
interaction hook. The player cannot read any of them. Three downstream
consequences:

1. The chapter opening loses the diegetic onboarding `story.txt` Beat 1
   describes (*"This opening should teach movement and interaction
   before dialogue dumps begin"*) — the player has nothing to interact
   with on the street except NPCs.
2. The Office Street English-first signage rule
   (`feedback_pig_swine_english_first_signage.md`) is unenforceable
   without canonical text — there is nothing to audit.
3. The Court Signpost in particular sits at the branch point and is
   the in-fiction reason the player walks north to the District Court
   in Beat 11. With no text on it, the route gate is purely a Sprite2D
   plus an off-screen `chapter1.court_unlocked` flag — the player has
   no way to read what the signpost is *for*.

Worth noting up front: the dialogue runner's per-NPC state-machine model
is not the natural fit for short single-line signage. Reading a notice
board is interaction, not dialogue. The notice-board text should pop in
the dialogue box (same UI), but it should live in its own data file
and be dispatched by an `interactable.gd`-style Area2D + label binding,
not by `dialogue_runner.gd` looking up per-NPC states. This keeps the
notice board from polluting the dialogue catalogue's per-NPC structure
and lets signs/notice text round-trip through the same pretty-print
contract as the rest of `data/`.

### Proposed addition

Design artifact (one sprint):

1. **`data/signage_ch1.json`** (new file, Design-owned). Schema:
   ```json
   {
     "version": 1,
     "_doc": "Chapter 1 Office Street signage text. Single source of
       truth for placed Sprite2D props that the player can interact
       with. Read by interactable.gd via sign_id binding. English-first
       per feedback_pig_swine_english_first_signage.md.",
     "signs": [
       {
         "id": "pig_swine_office_sign",
         "_anchor": "office_street.tscn::PigSwineSign",
         "lines": ["Pig & Swine.", "Attorneys, counselors, possibly open."]
       },
       {
         "id": "street_notice_board_ch1",
         "_anchor": "office_street.tscn::NoticeBoard",
         "lines": [
           "(a community notice board, three pinned items deep)",
           "• 'CAT MISSING. ANSWERS TO MOTION TO DISMISS.'",
           "• 'BUILDING WORKS, FLOOR 4, ENTRANCE STILL TECHNICALLY ALLOWED.'",
           "• 'TENANTS' ROW MONTHLY MEETING — INDEFINITELY POSTPONED.'"
         ]
       },
       {
         "id": "court_signpost_ch1",
         "_anchor": "office_street.tscn::CourtSignpost",
         "lines": ["District Court →"],
         "lines_unlocked": ["District Court →", "Open today. Try not to be late."],
         "_gate_flag": "chapter1.court_unlocked"
       },
       {
         "id": "street_name_office_street",
         "_anchor": "office_street.tscn::StreetNameOfficeStreet",
         "lines": ["OFFICE STREET"]
       },
       {
         "id": "street_name_counsel_row",
         "_anchor": "office_street.tscn::StreetNameCounselRow",
         "lines": ["COUNSEL ROW"]
       }
     ]
   }
   ```
   Source text drawn from `dialogue_samples.txt §Sign and notice text
   samples` register; English-first per
   `feedback_pig_swine_english_first_signage.md`; missing-cat-joke
   pre-cleared as Taste-Standard 5/5 (it's a real Warsaw kiosk-board
   register, anchored to a real procedure name, alive, clear, and the
   joke compounds — not contradicts — at a Ch5 missing-document beat).

2. **Code follow-up artifact (paired)** — a new
   `scripts/actors/sign.gd` (Code-owned), modelled on
   `scripts/actors/pickup.gd`. It takes `sign_id: String` as an exported
   property, looks up the entry in `data/signage_ch1.json` at boot,
   and dispatches the appropriate `lines` (or `lines_unlocked` when
   `_gate_flag` evaluates true) through the existing dialogue box on
   interaction. No state mutation. No `met_<x>` flag bookkeeping. The
   sign is `chapter1.court_signpost_read = true` if Design wants a flag
   for an Asia hint state callback, but optional.

3. **Scene wiring follow-up (Code-owned)** — add `Area2D` + interaction
   collider as a child of each of the five named Sprite2D nodes in
   `office_street.tscn`, attach `sign.gd`, set `sign_id` to the matching
   entry id.

### Acceptance criteria

- `python -m json.tool data/signage_ch1.json` passes.
- `node tools/verify_dialogue_roundtrip.js` passes (or whichever

- Beat 1 Asia hint state references the notice board as already-read

  `notice_board_read` flag by default to keep the proposal scoped.
- No string in `office_street.tscn` is player-facing text any more —
  re-asserts the AGENTS.md §Stack invariants invariant.

## Proposal 3: File the `smokers_lawyer_ch1` voice spec; ship a minimal stub to silence the runtime warning

**Type:** NARRATIVE (voice spec) + DATA (stub)
**Addresses:** the 2026-05-18 sprint placed a `smokers_lawyer_ch1` Area2D
NPC with sprite frames in `office_street.tscn` (line 314), but no voice
spec exists in `story.txt`, `style_canon.txt`, or
`data/voice_references/`. Per `.antigravity/skills/design.md`
§Halt conditions, writing dialogue for an NPC whose voice is not
established is a hard halt: *"file a voice-spec request artifact and
halt."* No such artifact exists. The character has been on the street
for six days unable to be authored.
**PROPOSALS.md status:** not mentioned. The character is implicit in the
Pixellab-pack 02 brief (`art/_pixellab_street_pack_02.md §D1 Smoking
junior lawyer`) — Art's spec — but Design has never written a voice
spec for him.

### Gap

`office_street.tscn` line 314: the `SmokersLawyer` Area2D references
`data/dialogues/smokers_lawyer_ch1.json` (because `npc.gd` routes by
`npc_id`), but no such file exists. The dialogue runner logs a
warning every time the player walks within interaction range of the
sprite. The SPRINT_LOG entry of 2026-05-18 explicitly flags this as a
"Design role follow-up" — six days later, nothing has been authored.

The hold-up is legitimate. Per `design.md`, Design cannot author a line
for an NPC whose voice is unspecced. The fix is to file the voice spec
first, then either author or stub.

Two related considerations argue for shipping a stub immediately even
before voice work lands:

1. The runtime warning has been firing for six days. Smoke tests
   currently filter or ignore it; that's a tolerated false-green.
2. The other four 2026-05-18 NPCs (`tram_waiter_ch1`,
   `mail_carrier_ch1`, `route_blocker_business_ch1`,
   `route_blocker_residential_ch1`) all have INERT skeleton drafts in
   `data/_drafts/` with `_doc:` placeholder lines. `smokers_lawyer_ch1`
   does not. It is the only one without even a skeleton — which is
   correct (no voice spec) but also visibly anomalous.

The character's function is clear from the Pixellab brief and the
scene placement: a smoking junior lawyer outside Pig & Swine; sidewalk
texture; not quest-critical; possibly transactional (a Polish-bar
register matter — interns smoke between hearings). Per
`AGENTS.md §First-meeting introductions` carve-out, transactional NPCs
do not require a name-exchange first-meeting beat. So a one-line
ambient interaction in the same shape as the `tram_waiter_ch1` draft is
the natural target — but the *voice* (cynical-junior? exhausted-
careerist? furtive-pleasant?) has not been specified.

### Proposed addition

Design artifact (split):

1. **`narrative_revision/voice_spec_smokers_lawyer_ch1.md`** (new file).
   Modelled on the per-character voice profile entries in
   `style_canon.txt §2`. Captures:
   - Who: a junior associate at a competing Office Street firm, or a
     trainee from one of the corporate towers further east. Specifically
     NOT Pig & Swine staff (the firm is a known quantity to the player;
     a hostile-but-fair external register is more useful here).
   - Why on this corner: the corner is one of the few smoking-tolerated
     spots in the legal quarter (real Warsaw-bar register; the smoking
     rules outside court buildings are a real procedural detail).
   - Register: dry, tired, slightly contemptuous of his own job; not
     hostile to Cula; willing to acknowledge a peer. One legal-shop-talk
     reference per line. NOT theatrical (that's Whimsy's lane).
     Inspirations push: less Hrabal-Pig, more Murrow-adjacent but
     without Murrow's archival precision — call it "junior-Murrow
     before the precision is earned."
   - Address form: toward Cula, *"Cula"* on professional recognition
     (the Polish-bar small-world register), or *"counsel"* if Cula has
     not yet entered the conversation. Not *"Dr. A. Cula"* — that
     register belongs to the firm and the client.
   - Forbidden: any reference to a real Polish firm; any malicious
     intent (per `AGENTS.md §Forbidden patterns`); any sex/scatological
     joke.
   - Out-of-scope: a recruitment arc, a Chapter-2-callback, or any
     mechanical interaction. He is a sidewalk fixture.

2. **`data/dialogues/smokers_lawyer_ch1.json`** (new file, MINIMAL
   stub). Single state, single line, marked `_voice_spec_pending`. The
   stub silences the dialogue_runner warning at boot and at
   interaction. Schema:
   ```json
   {
     "version": 1,
     "_voice_spec_pending": true,
     "_voice_spec_ref": "narrative_revision/voice_spec_smokers_lawyer_ch1.md",
     "_doc": "INERT STUB pending voice-spec acceptance. Promote when voice spec lands.",
     "npc_id": "smokers_lawyer_ch1",
     "display_name": "Smoking junior lawyer",
     "states": [
       {
         "id": "smokers_lawyer_ambient_stub",
         "trigger": "",
         "lines": ["(He glances up, registers a colleague, looks back at his cigarette.)"],
         "tags": ["chapter1", "ambient", "office_street", "stub"]
       }
     ]
   }
   ```
   The single line is a stage-direction-style observation, NOT
   dialogue. It is voice-neutral (deliberately) so it does not lock in
   a register before the voice spec lands. After voice spec acceptance,
   Design replaces this state with the authored lines, drops
   `_voice_spec_pending`, and the file becomes canonical.

### Acceptance criteria

- `python -m json.tool data/dialogues/smokers_lawyer_ch1.json` passes.
- Dialogue runner boot is clean — no missing-dialogue warning for

- Stub explicitly carries `_voice_spec_pending: true` so a future
