## 1. Retry-honor for `barista.json::coffee_retry_prompt`

Design has split `barista.json::coffee_outcome` into five buff-gated outcome states and authored a new `coffee_retry_prompt` state with an in-dialogue `options` block per `godot/CONVENTIONS.md` §Dialogue option schema. The retry state's trigger is `chapter1.coffee_buff != ""` (the design brief's specified gate); its options write the player's choice to `chapter1.coffee_retry_decision` with values `"retry"` or `"accept"`. Two pieces of Code wiring are still required for the retry to be reachable and behaviourally meaningful. First, the outcome states fire before the retry state in JSON order (the dialogue runner picks first-match), so on the first post-minigame interaction the player gets the outcome line, not the retry; on every subsequent interaction the outcome line fires again, indefinitely. The retry state can only become reachable when an outcome's `on_dismiss` flips an acknowledgement flag (e.g., `chapter1.coffee_outcome_acknowledged`) and the outcome triggers are extended with `&& chapter1.coffee_outcome_acknowledged == false`. Code needs to declare that flag in `State.reset_state()` and bump `SAVE_VERSION` with a v9→v10 migration. Second, `chapter1.coffee_retry_decision` is referenced by the options `write_path` but does not exist in `State.data` — the runner's `_set_state_value` silently no-ops on unknown paths, so the choice never persists. Code needs to declare that flag too, and add the actual retry behavior in `coffee_brewing.gd` (or its trigger): when `coffee_retry_decision == "retry"` is observed at trigger time, clear `chapter1.coffee_buff` / `chapter1.coffee_brew_grade` / `chapter1.coffee_tutorial_seen` / `chapter1.coffee_outcome_acknowledged` / `chapter1.coffee_retry_decision` and re-launch the minigame; when `"accept"`, just clear `coffee_retry_decision` so the prompt stops asking. The transient nature of the decision flag is up to Code — it can be reset to `""` after consumption, or held in `State.session_*` rather than persisted save state.

## 2. Engine reader for `data/minigames/coffee_text.json`

Design has authored a new file `data/minigames/coffee_text.json` containing player-facing strings the engine currently hard-codes: the three phase labels (`Grind.` / `Pour.` / `Serve.`), one flavor line per phase (the "Grinding beans. Please avoid creating procedural dust." family from `minigames.txt` §Phase 1–3), result lines keyed by buff value (the four `minigames.txt` §Core fantasy lines mapped onto `procedurally_alert_plus` / `procedurally_alert` / `caffeinated` / `over_caffeinated`), and the two final-stamp lines (`STAMPED: CAFFEINATED` / `STAMPED: QUESTIONABLE`). `coffee_brewing.gd::_update_phase_label`, `coffee_brewing.gd::_show_result`, and the stamp render path currently set these labels from string literals. Code's Prompt 8 should add a small loader (mirroring the existing `coffee_patterns.json` loader at `_load_pattern_from_json`) that reads `res://data/minigames/coffee_text.json` once on `_ready` and exposes a `text_for(slot: String, key: String) -> String` helper. The three render paths read from that helper instead of literals — phase-label set from `phase_labels[phase_name]`, flavor from `phase_flavor[phase_name]` (Code chooses whether to render this as a second Label below `PhaseLabel` or as the existing PhaseLabel's second line), result-panel detail from `result_lines[result.buff]`, and the stamp impression from `result_stamps["hit"]` when the final stamp prompt registered as a hit, `"missed"` when it missed. The file is loaded by no consumer until this is wired; the JSON is valid and Taste-Standard-passed, so the strings ship the day Code reads them.
